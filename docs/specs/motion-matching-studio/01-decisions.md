# MotionMatching Studio Decisions

Created: 2026-05-14

This document records accepted design decisions so future sessions can continue without relying on chat history.

## Product And Scope Decisions

- Product name: `MotionMatching Studio`.
- The Studio is a standalone authoring and preview tool.
- Unity must not be required to author or preview a character database.
- The Studio must be able to run in browser/dev mode inside Codex through a local URL.
- Electron packaging is planned later; MVP starts with Vite dev server plus ASP.NET Core backend.
- The Studio should eventually be usable as a single desktop app, but the architecture remains web-first.

## Core Architecture Decisions

- The original MotionMatching engine should evolve into shared core libraries plus host adapters.
- Unity is one adapter, not the center of authoring.
- Authoring can be rich and complex; runtime should stay small and portable.
- Database format and runtime contracts are the long-term portability boundary.
- C# core is the reference implementation, not necessarily the only future runtime implementation.
- .NET solution name: `MotionMatchingStudio.sln`.
- Intended source layout:
  - `src/MotionMatching.Core/`
  - `src/MotionMatching.Authoring/`
  - `src/MotionMatching.Importers/`
  - `src/MotionMatching.Builder/`
  - `src/MotionMatching.PreviewRuntime/`
  - `src/MotionMatching.Studio.Backend/`
  - `apps/studio-web/`
  - `tests/MotionMatching.Authoring.Tests/`
  - `tests/MotionMatching.Importers.Tests/`
  - `tests/MotionMatching.Builder.Tests/`
- Backend/core/importer/builder tests should be added from the start.
- .NET tests use xUnit.
- Frontend MVP starts with lint/typecheck; component/e2e tests can come later.

## Frontend And Backend Decisions

- Frontend: Vite + React + TypeScript.
- 3D viewport: Babylon.js.
- Backend: ASP.NET Core.
- Backend/core target framework: .NET 8.
- Future distribution should prefer self-contained backend binaries for macOS and Windows.
- Frontend runtime/tooling: Node 22 LTS and npm.
- Frontend package name: `motion-matching-studio-web`.
- Frontend styling starts with plain CSS, CSS variables, and `lucide-react` icons.
- Avoid a heavy UI component library for MVP.
- MVP layout uses resizable splitters for left panel width, right inspector width, and bottom area height.
- Full docking is deferred, but should remain a future design goal.
- Layout preferences such as splitter sizes are machine/browser-local, not workspace state.
- Theme preference is machine/browser-local, not workspace state.
- Selected character/clip is local UI state, not portable workspace state.
- The current character for build/play is derived from UI selection, not stored as workspace data.
- Backend API: REST for commands/state, WebSocket for build progress and preview frames.
- API JSON uses camelCase properties and snake_case semantic enum/string values, matching manifest conventions.
- REST API routes are versioned under `/api/v1`.
- WebSocket routes are versioned, for example `/ws/v1/tasks`, `/ws/v1/preview`, and `/ws/v1/logs`.
- Dev mode default backend port: `5100`.
- Dev mode default Vite port: `5173`.
- Dev CORS should allow `localhost:5173` and `127.0.0.1:5173`.
- Browser/Codex dev mode imports files through frontend upload to the backend.
- Electron/native desktop mode can later pass native file paths directly to the backend.
- MVP upload size limit is `50 MB`.
- If an upload exceeds the limit, the UI shows a clear error and can point users to future desktop/native import mode.
- Browser/Codex dev mode uses a repo-local hidden workspace at `.motionstudio/browser-workspace/` for easy Codex testing.
- `.motionstudio/` is machine-local/dev data and must be git-ignored during implementation.
- Browser workspace derived preview GLB/cache lives under that workspace's `Derived/Cache`.
- Operation logs live under the relevant workspace/character `Derived/Logs`.
- Operation logs are derived and should not be included in portable workspace zips by default.
- Backend server logs can live under `.motionstudio/logs/` in dev mode.
- Backend logging starts with built-in `Microsoft.Extensions.Logging`; Serilog or richer structured logging can be added later.
- Frontend must not parse FBX or BVH directly.
- Backend imports assets and produces web-friendly preview cache.
- Preview mesh format: GLB generated by backend from Visual FBX.
- Preview frame streaming can start as JSON transforms for MVP.
- No Blender dependency for MVP; backend should use libraries for import/conversion.
- Candidate import/conversion path: AssimpNet for FBX import and SharpGLTF for GLB export, subject to validation with real assets.

## UI Layout Decisions

- The UI should feel like a desktop authoring tool, not a dashboard.
- UI style should be compact, utilitarian, and editor-like.
- Avoid decorative flourishes, marketing-style cards, and unnecessary visual effects.
- Prefer dense but readable panels, clear controls, and low visual noise.
- Default MVP theme is a neutral dark editor theme.
- Status colors:
  - Error: red.
  - StrongWarning: orange.
  - Warning: amber.
  - Info: blue/gray.
  - Success/valid: subtle green.
- Left panel: workspace characters and their clips.
- Left panel uses a direct `Character -> clips` tree, without an extra `Characters` group level.
- Visual model is not a separate tree child; selecting the character shows visual/settings/validation in the inspector.
- Clips are flat children under each character in MVP.
- Role/tag folders or grouping are not required in MVP; use search/filter instead.
- Clip rows may show `clipRole` as a compact chip.
- Clip rows show validation badges/finding counts.
- Clip include/exclude from build should be accessible from the inspector and context menu; a compact row control can also be used if it does not clutter the tree.
- Clip rows expose a context menu for destructive clip actions. MVP deletion removes the clip from the character manifest and deletes the managed clip folder/cache from the browser workspace after confirmation.
- Clip settings are editable from the right inspector in MVP: include/exclude from build, one controlled `clipRole`, default global tags, and custom tags. These settings are persisted to the clip manifest through a dedicated backend settings endpoint.
- Center: 3D viewport.
- Right panel: universal context-sensitive object inspector.
- Bottom area: timeline and logs/validation visible together in MVP.
- Later, logs and validation may become tabs/drawers with error count icons.
- Importing or replacing an entity automatically selects that entity.
- The universal inspector always shows the currently selected entity context.
- After Visual FBX import, the character is selected, the viewport shows the visual mesh, and the inspector shows character/import/validation details.
- Selecting a character shows character, visual, build, and tuning settings in the inspector and displays the character in the viewport.
- Selecting a clip shows clip role, tags, import settings, contact settings, and findings in the inspector and previews the clip in the viewport.
- Changing selection labels, such as switching between a character row and one of its clips, must not unload and reload the same preview GLB. The viewport reloads the character preview only when the preview asset URL changes; rapid selection clicks should only update overlay text and inspector state.
- Current Clip Import MVP shows clip source metadata in the inspector; clip animation preview and findings are the next implementation slice.
- Clip manifests may store timeline metadata (`frameCount`, `frameRate`, `durationSeconds`) when the backend can extract it. BVH imports read this from the `MOTION` header. FBX clip imports use Assimp to export a temporary GLB and read animation sampler timing from glTF accessors. Because glTF channels may contain extra nonuniform keys on a few bones, the timeline parser chooses the dominant full-duration sampler count instead of treating an arbitrary channel key count as the clip frame count. If extraction fails, the frontend may show an estimated preview range until real metadata is available.
- FBX clip animation preview starts with a pragmatic cache path: backend exports the clip FBX to a derived GLB under the clip cache, exposes that asset URL in the clip API response, and the Babylon viewport retargets the imported animation group to the current visual skeleton by matching node/bone names. Retargeting applies rotation tracks broadly, filters out scale tracks, and only allows translation on root/hips-level targets so clip imports do not overwrite the visual skeleton proportions. Timeline scrub maps UI frames onto the loaded Babylon animation group's actual frame range. After an animation group loads, the viewport immediately applies the current timeline frame, including frame `0`, so the preview does not remain in the visual model's bind/T-pose until the user scrubs.
- The clip inspector should expose a concise animation preview/retarget status, including how many animation tracks were actually applied, so failed or unmatched retargeting is visible without opening developer tools.
- The clip inspector should expose read-only root motion diagnostics from the preview GLB: detected root motion source node, key count, source displacement X/Y/Z, horizontal distance, and average horizontal speed. These diagnostics explain what is actually inside the imported clip and must not change source data.
- The viewport snapshots the visual model's base character pose after loading the character preview. Selecting the character itself, clearing clip selection, or loading another clip restores that base pose first so old animation frames cannot remain stuck on the skeleton.
- Animation preview defaults to `In-place` motion. In this mode, retargeting locks root-motion translation on X/Z while preserving Y, and locks top-level root/skeleton rotation to the first key. This keeps locomotion clips inspectable without the character running out of frame while still showing vertical motion. A compact viewport toolbar toggle switches to `Root motion` when the full source root transform should be reviewed.
- `In-place` is strictly a viewport preview policy. It must not rewrite clip source files, imported manifests, derived GLB cache, or future build/runtime motion data. Root motion remains the source of truth for trajectory extraction, velocity/facing features, simulation-bone data, and any later runtime controller behavior.
- MVP clip skeleton validation compares normalized bone names from the selected character's Visual FBX against the clip source skeleton. FBX skeleton names come from `assimp info` hierarchy parsing; BVH skeleton names come from `ROOT` and `JOINT` declarations before `MOTION`. Validation initially reports unmatched skeletons as errors and low bone-name coverage or missing root/hips/pelvis candidates as warnings.
- Clip skeleton validation also exposes read-only match diagnostics in the inspector: visual bone count, clip bone count, matched count, coverage percentage, missing critical names, and small samples of matched/visual-only/clip-only names. These diagnostics are there to explain importer behavior and should not become editable mapping state in MVP.
- Import diagnostics are shown in the inspector as a compact import log. The backend response should include enough generated log entries to explain source load, timeline extraction, preview cache availability, root motion diagnostics, and skeleton validation without requiring developer tools.
- Validation findings shown in logs/validation panels are clickable.
- Clicking a finding selects the related entity and jumps the timeline/viewport to the affected frame when frame data exists.
- On startup, if a recent workspace exists, open the most recent workspace automatically.
- If no recent workspace exists, show a start screen with Create Workspace and Open Workspace actions.
- Recent workspace paths are machine-local settings, not portable workspace state.
- `New` creates an empty workspace; it does not require selecting a Visual FBX immediately.
- In browser/Codex dev mode, `New` may create a browser/test workspace flow suitable for local testing before full desktop file-picker integration.
- In browser/Codex dev mode, if a browser workspace exists, open it by default.
- Empty workspaces show `Add Character` in the left panel.
- The 3D viewport shows an empty scene for empty workspaces, not a large onboarding card.
- `Add Character` opens a Visual FBX file picker and creates the character from the selected file.
- There is no empty character without a Visual FBX in MVP.
- The first real local test asset is an Iyo Mixamo FBX supplied by the user. Its absolute machine-local path must not be written into portable docs/manifests.
- The Studio UI should stay close to a practical editor: no landing-page composition, decorative tool panels, or placeholder-heavy areas. The bottom timeline/logs area remains part of MVP, but should stay compact until playback, validation, and task controls are implemented.

## Viewport Decisions

- Center viewport supports two modes:
  - Animation Preview Mode: scrub/play selected clip on canonical visual mesh.
  - Play Mode: flat-plane mechanics preview with selected character.
- MVP should render the skinned visual mesh, not only bones.
- Skeleton/bone display can be a debug overlay.
- Viewport supports perspective and orthographic camera types.
- Switching between perspective and orthographic preserves the apparent viewport scale; orthographic bounds are derived from the current camera radius and perspective FOV.
- Viewport should include an orientation helper, preferably camera axes similar to Blender or a camera cube.
- The Studio viewport follows the Babylon world convention used by the ground grid: `X` is red/right, `Y` is green/up, and `Z` is blue/depth.
- The orientation helper is a compact clickable axis gizmo:
  - `X` / `-X` switch to right / left views.
  - `Y` / `-Y` switch to top / bottom views.
  - `Z` / `-Z` switch to front / back views.
  - The center dot switches to an isometric view.
  - The perspective/orthographic toggle lives directly under the axis gizmo.
- Viewport navigation:
  - WASD moves the camera.
  - Shift doubles camera movement speed.
  - Mouse wheel zooms.
  - Middle mouse drag pans.
  - Left mouse drag orbits.
  - `F` frames the selected character/clip.
  - Numpad `1`/`3`/`7` switch to front/right/top views.
  - Numpad `5` toggles perspective/orthographic.
- The viewport ground grid treats the current source scene as centimeter-like units for Mixamo-style assets: minor grid lines are every `10` units and emphasized meter lines are every `100` units. Runtime scale normalization is still handled by build settings; the viewport grid is a visual authoring aid.
- Add context menus where useful.
- Context menus should be available where natural:
  - Character rows: Rename, Replace Visual FBX, Build, Export, Validate, Delete.
  - Clip rows: Rename, Replace Source, Include/Exclude from Build, Reprocess, Validate, Delete.
  - Viewport: Frame Selected, Reset Camera, Toggle Skeleton, Toggle Foot Contacts, Switch Camera Mode.
  - Timeline: Go to Frame, Play/Pause, Step, later add/edit range tags.
- Viewport toolbar provides mode controls, including Preview Mode and Play Mode.
- `Ctrl+G` toggles Play Mode.
- `Esc` exits Play Mode.
- MVP preview materials can be simple/basic.
- Textures should be displayed when available, but render fidelity is not the priority.
- Viewport material modes:
  - Textured
  - Flat
  - Shaded
- MVP debug toggles:
  - Show Skeleton
  - Show Foot Contacts
  - Show Root Motion Trail
  - Show Trajectory Query
  - Show Selected Pose Info
  - Material Mode: Textured / Flat / Shaded
- Play Mode debug data is shown in two levels:
  - Compact HUD overlay in the viewport.
  - Detailed state/pose data in inspector/log panels where needed.
- Play Mode debug HUD includes:
  - State.
  - Current clip role/name.
  - Current frame.
  - Speed.
  - Target speed.
  - Grounded/Airborne.
  - Jump committed direction.
  - Selected pose index.
  - Warnings such as missing role or stale build.

## Workspace And Filesystem Decisions

- Project state is filesystem-based JSON, not SQLite.
- Workspaces are self-contained and portable as a zip.
- Imported source assets are copied into the workspace.
- Build artifacts and preview/import caches are derived and disposable.
- Do not store build artifacts or cache as required portable state.
- Use human-readable folders for people and future agents.
- Entities have stable short prefixed IDs plus human-readable names/folders.
- ID format uses a short prefix plus 8 lowercase random base32/base36-style characters, for example:
  - `char_ks8d3h2a`
  - `clip_m4q9x1bp`
  - `tag_v2p8n7kc`
  - `task_t9k1a0rm`
- JSON manifests are pretty-printed, stable ordered, 2-space indented, and newline-terminated.
- JSON serialization uses `System.Text.Json` on the backend.
- JSON schema validation uses `JsonSchema.Net`.
- JSON property names use `camelCase`.
- Serialized semantic enum/string values such as `clipRole` use `snake_case`.
- Every manifest includes `schemaVersion`.
- JSON schemas are required.
- Schemas live in the tool repository under `docs/schemas/`.
- Manifests should include `$schema` when practical.

## Replacement Decisions

- Replacing Visual FBX overwrites `Visual/source.fbx`.
- After Visual FBX replacement, the canonical skeleton/cache is recalculated, clips are revalidated, and the character becomes dirty.
- Existing clips remain after Visual FBX replacement; validators report issues and the user decides.
- Replacing clip source overwrites the managed clip source file.
- Replacement preserves clip ID, clip name, tags, settings, and include flag.
- Clip source replacement is exposed from the clip inspector as `Replace Source`. It accepts FBX/BVH, clears derived clip cache, reparses timeline metadata, regenerates preview/diagnostics on demand, and preserves include/mirror/contact preset/role/tag settings.
- Character rows expose a context-menu delete action. Deleting a character removes the workspace reference and the character folder, including clips and derived caches, after explicit confirmation.
- If extension changes, the managed source filename extension updates.
- Timeline/range tags transfer by frame number.
- There is no built-in source revision history in MVP.
- Destructive actions such as Delete Character and Delete Clip require confirmation dialogs.
- Delete Clip physically removes the clip folder from the workspace.
- Delete Character physically removes the character folder from the workspace.
- There is no soft delete/trash in MVP.
- Rename Character renames the character folder and preserves the stable character ID.
- Character rename conflicts use suffixes such as `_001`.

## Character And Skeleton Decisions

- In MVP, creating a usable character starts with importing a Visual FBX.
- Visual FBX is required before importing animation clips for that character.
- Visual FBX is the canonical character source.
- Visual FBX provides:
  - Mesh
  - Skin weights
  - Bind pose
  - Canonical skeleton
  - Preview materials/textures where available
- If Visual FBX contains multiple skinned meshes or material groups, MVP imports all parts bound to the canonical skeleton as one visual model.
- Visual FBX must contain exactly one character skeleton.
- Visual FBX files with multiple independent skeleton roots/armatures are rejected in MVP.
- Animation FBX/BVH files are motion sources only.
- Animation source meshes/materials are ignored for normal preview.
- Animation source files must contain exactly one animation skeleton in MVP.
- Animation source files with multiple independent skeleton roots/armatures are rejected in MVP.
- Any animation preview is shown only on the canonical Visual FBX mesh.
- Source animation skeletons are import-time details.
- All animation sources are mapped/retargeted onto the canonical visual skeleton before preview, tagging, feature extraction, and database build.
- MVP avoids manual bone mapping as the primary workflow.
- The importer should guess mappings programmatically.
- If important mapping is uncertain, validators report findings; user can inspect preview and decide.

## Bone Mapping Decisions

MVP automatic bone mapping strategy order:

1. Exact bone name match.
2. Normalized name match, such as removing prefixes, namespaces, suffixes, case, spaces, underscores, and separators.
3. Known synonym rules, such as pelvis/hips and thigh/upperleg.
4. Hierarchy-aware and side-aware guesses for left/right limbs.
5. Confidence score per mapped bone.

Mapping guess rules must be extensible through data/configuration where practical, not buried as hardcoded one-off logic.

Mapping rules have two intended levels:

- Built-in rules shipped with the tool.
- Optional workspace/character override rules later.

MVP should start with built-in JSON/config-based rules where practical. Character override rules can come after MVP.

Mapping UI should show a concise summary and problematic/low-confidence bones by default.

A full mapping table can be available in a collapsible advanced section.

MVP mapping details are read-only. Manual per-bone mapping overrides are not required for MVP.

If auto mapping is wrong in MVP, the user fixes the source asset, replaces the clip, or adjusts import settings. Manual bone override is deferred.

Future retargeting fixes should be guided by a wizard rather than defaulting to a large manual mapping editor.

The retargeting fix wizard is not required for MVP; it is a Phase 2 workflow.

The intended wizard flow:

1. User reports or selects "I see a problem".
2. The wizard asks which bone/body area appears wrong.
3. The wizard asks what kind of problem is visible.
4. The wizard offers likely fixes based on validators, mapping confidence, and known problem patterns.

Known retargeting problem types should grow over time from real assets and validator findings.

Animation import settings should be guessed automatically and editable in the inspector:

- Unit scale: auto/manual.
- Up axis: auto/manual.
- Forward axis: auto/manual.
- Root motion source: auto/manual.

The tool should apply reasonable guesses on import and let the user correct them.

Default root motion source priority:

1. Explicit root bone above hips with motion.
2. Hips/pelvis.
3. BVH root joint.
4. Manual override in inspector.

Changing expensive import settings should schedule background reprocess tasks instead of freezing the UI.

The UI should expose a task list/queue for imports, reprocessing, validation, and builds.

For MVP, the task list can live in the bottom logs area.

Later, a compact top/right task indicator can show running and queued counts.

Background tasks should support cancellation:

- Queued tasks are cancelable.
- Running tasks are best-effort cancelable when the underlying operation supports it.

MVP uses one global background task queue for the active workspace.

Tasks must avoid conflicting concurrent operations on the same character or clip.

MVP executes background tasks sequentially.

Parallel import/build/reprocess can be considered later after cache and manifest behavior is stable.

When clip import settings change, the viewport can continue showing the previous processed result while a background reprocess runs.

Stale previews must be clearly marked with a warning until the new processed result is ready.

After clip reprocess/replacement:

- Timeline/range tags transfer by frame number.
- Out-of-range tags produce warnings.
- `clipRole` is preserved after normalization; retired legacy roles such as `start`/`stop` become unassigned.
- Global tags are preserved.
- Auto suggestions may be recalculated, but accepted/manual user choices must not be overwritten silently.

## Validator Decisions

- Validators are separate classes.
- Each validator reports what it dislikes as findings.
- Findings include severity, message, affected bones/frames where practical, and suggested action where practical.
- Severity levels:
  - `Info`
  - `Warning`
  - `StrongWarning`
  - `Error`
- Validator findings do not automatically decide build inclusion.
- The user owns build inclusion through an explicit per-clip flag.
- Clips with warnings remain visible and previewable.
- Clips can be included or excluded from build by the user.
- New imported clips default to `includeInBuild = true`.
- Included clips with `Error` findings block build until fixed or excluded.
- Clips can opt into `mirrorInBuild`, shown in UI as `Add mirrored copy`. This is a build-time augmentation flag: build planning expands one included source clip into an original entry plus a mirrored entry. It is not a tag and does not create a second imported clip in the authoring tree.
- Character inspector shows a read-only Build Readiness panel before full database building exists. It lists included clips, planned mirrored copies, role coverage, warning/error counts, and the exact findings that would affect build readiness.
- First Build Plan is a report, not a runtime database artifact: it expands included clips into source/mirrored entries and summarizes role coverage, skeleton coverage, and foot contact presence/missing status.
- Build Readiness findings and Build Plan rows that refer to a clip are clickable and select that clip in the editor. Later frame-specific findings can extend this behavior by also seeking the timeline.
- `Generate Build Report` writes the current first build plan to `Builds/<CharacterName>/build-report.json`. This is a derived workspace artifact and does not produce runtime database files yet.
- Existing build reports are discoverable from the character response as `buildReportPath`; `View Report` can load the saved JSON after a browser reload.
- Build reports store a readiness fingerprint. Character response exposes `buildReportStatus` as `none`, `current`, or `outdated` by comparing the saved fingerprint with the current Build Readiness plan.
- `Build Runtime` writes `Builds/<CharacterName>/runtime-build-draft.json` plus first JSON drafts for `Builds/<CharacterName>/<CharacterName>.mmskeleton`, `.mmpose`, `.mmfeatures`, and `.mmdatabase`. It is the first runtime-facing build artifact: the draft lists the built-in feature preset, planned outputs, skeleton bone/slot data, planned pose samples per source/mirrored build entry, sampled local pose bone values from the clip preview GLB when available, planned feature channels/samples, and a combined database draft with clips, dense sampled feature values for every sampled frame, pose samples, scale metadata, and foot contact tracks. Runtime draft generation has a `sampleFrameStep` parameter from the character inspector so dense clips do not have to sample every frame. `.mmfeatures` keeps a rough feature value preview for the first sampled frames, while `.mmdatabase` stores the full sampled list and embeds `poseSamples` so the database draft is closer to a single runtime import payload. Preview feature values are still approximate: `hips_velocity` comes from root-motion diagnostics, foot velocity channels use foot-contact diagnostics (`0` while in contact, contact threshold while outside contact), and trajectory channels estimate future scalar displacement/direction for the configured future frame offsets. Scale is explicit because runtime engine queries must use the same units as the generated database. The character inspector exposes `Runtime scale mode`: `Auto`, `Source x0.01`, and `Character x1`. `Auto` warns and applies `0.01` if root speed looks like source-scale units (`> 20 u/s`); forced modes record the chosen mode in the draft so integration can match engine query units deliberately. It exposes `runtimeBuildDraftPath` and `runtimeBuildDraftStatus` (`none/current/outdated`) in the character response and can be loaded in a modal Runtime Build Preview. It is explicitly not the final Unity-compatible database serialization yet.
- Character inspector also shows the current effective runtime scale from the last loaded/generated runtime draft, for example `Source x0.01 x0.01` or `Character x1 x1`; if no draft is loaded, it shows `Not generated`.
- Runtime build settings (`sampleFrameStep`, `scaleMode`, `trajectoryPredictionFrames`) are stored in `character.json` under `runtimeBuildSettings` and updated through the character inspector, so they survive browser reloads. `trajectoryPredictionFrames` are sampling-frame offsets at the runtime sampling rate; the default `[10, 20, 30]` at 30 fps means `+0.333s`, `+0.667s`, and `+1.000s`. Feature channel metadata also carries those computed seconds so runtime/game queries can stay time-based even when source clips have different original FPS. Expanded trajectory value keys include the sampling-frame suffix, for example `trajectory_direction_10f`, `trajectory_direction_20f`, and `trajectory_direction_30f`; the `f` suffix means sampling frames, not source clip frame indices. The inspector shows a small save state while those settings are written. The Runtime Build Preview includes `Copy Query Contract`, which copies a compact JSON contract for engine integration: scale mode/factor, expanded feature layout, skeleton slots, and planned sample counts. Build Report and Runtime Build Preview also include `Copy Paths` buttons for the generated artifact paths. The character inspector has `View Database Draft`, which opens the `.mmdatabase` draft directly and can copy that database JSON.
- Runtime/game integration must read the exported query contract or database feature layout instead of hardcoding trajectory offsets. Studio owns the names and timing of compiled feature vectors; the engine fills query values by exact exported names and uses the exported seconds for lookahead timing.
- `.mmdatabase` contains both `schemaVersion` and a lightweight `schema` block (`id`, numeric `version`, `format`, `units`) so engine-side import can validate the file before consuming samples.
- Runtime Build Preview and Database Draft show a warning if feature/database samples exist but no pose values were sampled. The character inspector also has `Copy Build Folder`, which copies the portable build folder path such as `Builds/IyoMixamo`, and `Export ZIP`, which writes `Builds/<CharacterName>/<CharacterName>-runtime-build.zip` from the generated build folder. The ZIP excludes prior ZIP files and stores portable relative entries only.
- Database Draft includes a sample table with clip/frame filters and a Runtime Query Preview. Selecting a database sample shows the exact feature vector, scale metadata, source/mirrored clip identity, frame, and time, and can copy a compact query-sample JSON for engine-side integration checks.
- Clip preview uses each source clip's original timeline metadata (`frameRate`, `frameCount`, `durationSeconds`) when available. Sampling authoring uses its own authoring frame rate, currently 30 fps by default. Runtime/database build must convert source clip frames to real time with the source clip FPS (`sample.seconds = frame / clip.frameRate`, falling back to duration when FPS is missing), and database clip entries carry the source timeline metadata so engines can audit the frame-to-time conversion. Different clip FPS values are valid as long as matching/runtime logic compares sample times and feature horizons in seconds, not raw frame numbers.
- The UI should make this distinction visible: Sampling trajectory rows use `F@30` labels plus seconds, the Sampling timeline title includes the authoring FPS, matcher rows show source sample seconds next to source frames, Database Draft clip rows show source FPS/frame count/duration, Runtime Build Preview shows the expanded Query Contract, Sampling shows current query feature values by contract name, and the copied runtime query contract includes `samplingFrameRate`. The Sampling timeline has a `Query vectors` toggle that draws current-frame lookahead vectors in the 3D viewport using the exported trajectory horizon frames when a runtime draft is loaded.
- Sampling authoring is a milestone before runtime game mode. Sampling queries are named per-character authoring objects stored in `character.json` under `samplings`, and the workspace tree shows them as a sibling group to `Clips`. The Sampling tab is a visual query-builder/debugger for the future motion matching runtime: it shows the selected sampling's character, target role filter, capsule, facing direction, velocity, and future trajectory points in the same 3D viewport used by preview. The default sampling velocity is `[0, 0, 100]` in source centimeters per second, and the default trajectory points are projected from that speed over the 30 fps sampling horizon. Velocity is derived from the first trajectory point, so `Velocity X/Z` is displayed read-only and matcher/save payloads recompute it from the first point instead of letting it drift away from the drawn trajectory. The inspector can edit capsule size, facing angle, and trajectory frame/position values, including adding/removing points, as a live unsaved draft. The 3D viewport can drag trajectory points, the facing handle, and capsule height/radius handles into that same draft; trajectory points show `+Nf` labels so the prediction horizon is visible in the viewport. `Save Sampling` is the explicit persistence boundary. The Sampling tab also shows a first top-N matcher preview against the current runtime `.mmdatabase` draft by comparing the authored query features to database sample features. When the sampling has a target role, matcher preview first filters database samples to clips with that `clipRole`; `Any role` is kept for debugging broad searches. Matcher rows show normalized RMS score breakdown for velocity, trajectory position, and trajectory direction, so the values stay readable instead of exposing raw squared error sums. Single-clicking a matcher row selects it and draws a lightweight ghost pose in the sampling viewport when sampled pose values are available; double-clicking opens that source clip and seeks the animation timeline to the matched frame. Mirrored matches currently jump to the source clip/frame until mirrored preview playback exists. The ghost pose starts as a point-cloud plus inferred humanoid bone-line debug overlay because runtime pose samples currently lack explicit skeleton parent hierarchy; it should become a proper stick skeleton after hierarchy data is exported. Generated clip recipes and export come after this query/search surface is visible and agreed.
- Sampling ghost pose debug draws sampled glTF local poses in the centimeter viewport by scaling glTF meter translations by `100`, resolving common humanoid aliases such as `UpperChest`, `Left_UpperArm`, and `Left_LowerArm`, building readable torso/arm/leg chains from local translations/rotations, and remapping glTF Z-up pose positions into the viewport's Y-up scene. A selected matcher row draws a current ghost at the sampling origin for `matchedFrame`, plus one future ghost per trajectory point using pose samples at `matchedFrame + point.frameOffset`; each sampled hips is anchored horizontally to its trajectory point and rotated toward the trajectory segment with the current source forward-axis correction. It remains a debug overlay, not final retargeted generated-animation playback.
- The shared bottom timeline is reused for Sampling preview. In Sampling mode it ranges from frame `0` to the last trajectory point, shows trajectory point markers, and becomes active after samples are loaded from a runtime draft and at least one matcher result exists. If no matcher row is selected, the preview uses the first matcher result so the timeline can be scrubbed immediately after `Load Samples`. Scrubbing it displays one interpolated ghost pose at `matchedFrame + previewFrame`, wrapping the preview frame inside the matched source pose sample range when the trajectory horizon is longer than the source clip, anchored to the interpolated trajectory position and aimed along the current trajectory segment.
- Sampling timeline ghosting keeps the current preview skeleton visible by default and adds an optional semi-transparent trail from the bottom timeline combo: `none`, `every 10`, `every 5`, `every 2`, `every 1`, or a local `-5..+5` frame window around the current preview frame. Trail ghosts are smaller and more transparent than the current frame ghost.
- Sampling matcher query features are generated from the actual trajectory point frame offsets, not only the original `20/40/60` defaults. Matcher Preview is shown per trajectory point: five authored trajectory points produce five matcher rows, each row showing the best database sample for that point's velocity/trajectory-position/trajectory-direction query. If the current runtime database does not contain an exact trajectory feature channel for an authored point, Matcher Preview estimates the sample-side trajectory value from the available database trajectory offsets so added points still influence preview scoring before a future per-sampling feature build exists.
- Sampling trajectory rows show computed speed in `cm/s` for the outgoing segment from that row to the next point in frame order: distance to the next point divided by time to the next point. The trajectory list includes a read-only `0f` origin row with position `[0, 0, 0]`; its speed is origin-to-first-point speed so the start state is visible next to authored points without being saved as a real trajectory point. The last point shows `0 cm/s` until an explicit next point exists. The first authored point still defines displayed sampling velocity through origin-to-first-point projection; editing point frame offsets immediately changes segment speeds even if the row order stays the same. The last point is not implicitly a stop; a stop should be represented by point spacing/timing now, and can become an explicit end-state option later.
- The sampling viewport also writes outgoing segment speeds as small `cm/s` labels at the origin and trajectory points. This is a temporary readable visualization before richer color/arrow speed visualization exists.
- Trajectory rows have an insert action, and the 3D sampling preview exposes pickable insert regions along trajectory segments. Left-dragging a 3D trajectory point moves it; right-clicking a trajectory segment or point opens a small trajectory context menu instead of immediately mutating data. Segment menu insertion uses the projected click point for position and interpolates frame offset between neighboring segment endpoint frames; point menus offer insert-after and delete-point actions. Row insertion still uses midpoint frame/position for the following segment; if there is no following point, it extrapolates the last segment forward, or uses the sampling facing direction for a single-point trajectory.
- Right-click context selection in the sampling viewport uses screen-space nearest-point fallback before segment picking, so delete/point actions remain easy to hit in both perspective and orthographic cameras.
- Right-clicking empty ground in the sampling viewport opens the insert menu for the nearest trajectory segment using the clicked ground position as the insertion pick, so users do not need to hit the visible segment mesh exactly.
- The sampling viewport opens its right-click trajectory context menu on mouse-up only when the pointer did not move beyond a small click threshold; right-button drag remains camera pan.
- Later, sampling authoring should expand from a single runtime-style query into a generated-clip authoring workflow for graph-driven pipelines. The planned progression is:
  - Keep `Target role` on the sampling as the default role filter.
  - Add optional per-trajectory-point `roleOverride`, where `inherit` means use the sampling's target role.
  - Show per-point matcher candidates so the user can judge quality at each key point.
  - Add `Generate Clip Preview`, which stitches/plays the sequence of chosen matches as a generated clip preview.
  - Export the generated result as ordinary clips for non-motion-matching or graph-driven animation pipelines after the preview quality is acceptable.
- Clip rows and inspectors show validation state using colors and icons for errors/warnings.
- Left panel rows show aggregate validation badges.
- Character rows show highest severity and counts across visual import, included clips, and build readiness.
- Clip rows show the clip's highest severity and finding counts.
- Clips can be replaced with a new FBX/BVH source.

## Tags And Roles Decisions

- Do not mix tool state and engine semantics.
- Build inclusion is a tool/project flag, not a tag.
- A clip may have multiple global tags.
- Tags are drawn from a character-level vocabulary.
- Users can add custom vocabulary tags.
- Clip tags are edited as checkboxes in the right inspector.
- Add Tag expands the character vocabulary.
- Tags help filtering, search, and authoring metadata.
- Tags describe style, mood, performance variant, or context such as calm, happy, angry, tired, injured, combat, stealth, relaxed.
- Tags must not duplicate action semantics such as walk/run/turn/start/stop. Walk/run/turn/jump/fall/land belong in the controlled role vocabulary; starts/stops are treated as motion phases/events rather than standalone MVP roles.
- The clip inspector includes a helper to clear legacy action tags (`idle`, `walk`, `run`, `turn`, `start`, `stop`, `jump`, `loop`) from existing clips after the role/tag split.
- `disabled` is not part of the default semantic tag vocabulary.

Default global tags for MVP:

- `neutral`
- `calm`
- `happy`
- `angry`
- `sad`
- `tired`
- `injured`
- `combat`
- `stealth`
- `relaxed`

No generic `transition` tag in MVP. Action categories such as turn, jump, walk, run, fall, and land should be represented by `clipRole`, not tags. Start/stop are transition phases/events, not active roles.

Clip role is separate from tags:

- One clip has one controlled `clipRole`.
- `clipRole` drives preview/runtime semantics.
- Tags remain supplemental style/mood/variant metadata.
- PreviewRuntime uses role-constrained state transitions/search for gameplay states such as jump and land.
- For example, pressing jump chooses/searches within the appropriate jump role subset rather than searching the whole database blindly.
- Ground locomotion also uses role-constrained state transitions/search:
  - No input uses `idle_loop`.
  - Run input uses `run_loop`.
  - Shift-walk input uses `walk_loop`.
  - Starting/stopping are detected from trajectory/phase timing rather than separate `clipRole` values.
  - 180-degree turn input uses turn roles where applicable.

MVP turn testing focuses on 180-degree turns:

- Standing/in-place 180 turn.
- Moving/running 180 turn.
- 90-degree turns are not required for MVP.
- 180 turns are time-based committed actions, not instant facing flips.
- Turn duration is a character tuning parameter.
- MVP default turn durations:
  - `standingTurn180Duration = 0.45s`
  - `runTurn180Duration = 0.60s`
- MVP 180-turn tuning also includes:
  - `runTurnPivotTimeNormalized = 0.5`
  - `runTurnMinSpeedFactorAtPivot = 0.25`
- These defaults should be reviewed against implementation/code feel later.
- The inspector should show a simple curve preview for run-turn speed over normalized turn time.
- During committed 180 turns, input does not interrupt the turn.
- Turn input buffering is off by default.
- Run 180 turn follows the GDD intent: a committed deceleration -> pivot -> reacceleration action where momentum is preserved across the pivot.
- MVP can approximate the run-turn velocity/rotation curve using normalized turn time and tuning values.

MVP `clipRole` vocabulary:

- `idle_loop`
- `walk_loop`
- `run_loop`
- `turn_left`
- `turn_right`
- `turn_left_180`
- `turn_right_180`
- `jump`
- `fall_loop`
- `land`

Legacy role tokens from earlier experiments are normalized on save/load where possible: `start`/`stop` and old walk/run start/stop variants -> unassigned, `turn_180`/`run_turn_180` -> `turn_right_180`, jump variants -> `jump`, and land variants -> `land`.

Auto-detection should suggest both global tags and `clipRole`; the user can override `clipRole` with a combobox.

Animation clip import automatically runs mapping, validation, role/tag suggestions, and foot-contact detection.

After clip import, the clip is selected, the viewport previews it on the canonical visual mesh, and the inspector shows role/tags/contact settings and findings.

Clips without `clipRole` may still be included as generic search data with warnings. Character-level validators ensure required roles exist for Play Mode readiness.

## Auto-Tagging Decisions

- Prefer programmatic guesses and suggestions whenever possible.
- The tool should get the user to a working result quickly.
- Manual edits should correct guesses, not fill everything from scratch.
- Auto-tagging runs on import/reimport.
- Auto-tagging should not overwrite accepted/manual user edits without clear user action.
- Reasonable auto tags may be applied by default and remain easy to remove through checkboxes.
- Auto-tag thresholds for global locomotion speed:
  - `idle`: average root speed < 0.2 m/s
  - `walk`: 0.2-2.2 m/s
  - `run`: > 2.2 m/s
- Auto-tagging should add `loop` when a clip appears cyclic/stable, using conservative heuristics such as first/last pose similarity and stable root speed.

## Foot Contact Decisions

- MVP auto-detects foot contacts and visualizes them on the timeline.
- Foot contacts are initially detected by toes/foot velocity threshold with smoothing, matching the original engine spirit.
- Height-based refinement is planned later.
- Contact velocity threshold is a character-level setting with default `0.15`.
- Browser MVP foot-contact preview is read-only diagnostics derived from the clip preview GLB. It reconstructs foot/toe world-space motion from the glTF node hierarchy plus animated translation/rotation/scale channels, exposes the source node names in the inspector, and draws left/right contact bars on the timeline. Until asset unit normalization lands, the browser GLB diagnostics use a source-unit velocity threshold (`15 u/s`) rather than the future character-level normalized default; if that fixed threshold finds nothing, the preview falls back to a low-percentile velocity cutoff for that foot/toe track so the user still gets useful visual hints. This is preview/debug data only and must not rewrite clip source files or manifests.
- Foot-contact API responses expose range seconds from the source GLB and range frames normalized back onto the clip timeline. Do not show raw foot-track sample indices as clip frames, because foot/toe channels can have different key counts and sampling than the main clip timeline.
- The viewport also shows active foot contacts in 3D as transient colored markers near the matched foot/toe target: left is blue and right is amber. These markers are a preview overlay driven by the current timeline time and the read-only diagnostics, not authored clip data. Do not apply an extra visual frame bias after normalizing contact ranges onto the clip timeline; previous `+1` preview bias made contact starts appear one frame late.
- Foot contact diagnostics must discard a single contact range that covers almost the whole clip for locomotion clips. This usually means the detector has too little useful planted-foot motion signal for the current clip/scale settings, and showing it would mislead the user. Idle/stationary clips are the exception: `idle_loop`, `idle` tag, or near-zero root horizontal speed can legitimately have full-clip foot contacts.
- Foot contact diagnostics close a single-frame internal gap between contact ranges. Tiny one-frame velocity spikes can split one visually continuous plant into ranges like `F 1-8` and `F 10-11`; the preview should show that as one planted-foot contact instead.
- Foot contact diagnostics can include up to two touchdown lead frames before a detected planted range when the foot/toe is already near the planted height and descending. This keeps the preview aligned with visual ground contact without applying a blind frame bias to every range.
- Foot contact diagnostics keep representative foot and toe tracks for a side when they carry useful timing. A `Right_Foot`/`Left_Foot` track represents heel/foot strike, while toe tracks can still represent toe contacts and toe-off. If both are active, viewport marker selection prefers the foot track first.
- The clip inspector exposes a Foot Contacts refresh action. In the current MVP this is an explicit re-read/re-detect of read-only diagnostics from the derived preview GLB and returns an updated clip response; it must not persist authored contact ranges or rewrite the clip manifest until authored contact editing exists.
- Clips can override contact detection settings.
- Per-clip contact detection uses a preset combobox instead of exposing raw threshold/smoothing fields. Clip manifests persist only the preset token; the backend maps presets to detector parameters so future detector changes do not rewrite project data around old magic numbers.
- Initial contact detection presets:
  - `auto`
  - `source_scale`
  - `character_scale`
  - `strict`
  - `loose`
  - `manual_only`
- MVP implements the presets as velocity-threshold variants over the current read-only diagnostics. `manual_only` disables auto-detected preview contacts until authored contact editing exists.
- MVP visualizes detected contact ranges but does not require range editing.

## Timeline Decisions

- Timeline shows frames and seconds.
- Timeline stores the current frame as a zero-based frame index internally, clamps it to `0..frameCount-1`, and displays it to users as `Frame 1/<frameCount>` through `Frame <frameCount>/<frameCount>`.
- Timeline ruler should be visually frame-based, not only time-based: show per-frame ticks when practical, major frame labels, and a frame number on the playhead. Contact bars should draw from normalized `startFrame..endFrame` on the clip timeline so frame ranges are readable even when source GLB channel seconds are fractional inside a frame.
- Loop preview should show a seam hint for contact ranges that start on frame `0`: draw a one-frame contact mark at the final timeline frame and allow the 3D marker to activate there. This is only a browser preview continuity hint for looped clips and must not mutate the raw detector ranges.
- Timeline scrubber supports mouse/pointer drag with pointer capture, not only click-to-seek, so users can continuously scrub animation frames in the viewport.
- Timeline playback is driven by wall-clock elapsed time through `requestAnimationFrame`, but discrete preview frame selection follows the clip frame-rate cadence: frame `n` is held for `1 / frameRate` seconds and the loop length is `frameCount / frameRate`. Source duration metadata still describes the animation time range between first and last keys, but it should not make the final displayed frame vanish during looped editor preview.
- Timeline loop range is editor-only preview state. It is represented by zero-based start/end frames internally, displayed as one-based `In`/`Out` values, and affects playback looping only; it must not rewrite clip manifests or source animation data.
- Timeline should expose direct frame entry plus quick first/previous/play/next/last controls so testing exact frame poses does not require drag-only interaction.
- Timeline has Play/Pause.
- Timeline has step -1 frame and +1 frame.
- Until imported clip duration/frame metadata is exposed by the backend, the frontend timeline may use a temporary fixed preview range as a UI scaffold only.
- Keyboard shortcuts should support play/pause and frame stepping.
- MVP shortcuts:
  - Space: Play/Pause
  - Q: previous frame.
  - E: next frame.
  - Left/Right: step one frame when paused.
  - Shift+Left/Right: jump 10 frames
  - Home/End: first/last frame

## Play Mode Decisions

- Play Mode runs only the currently selected character.
- In Animation Preview Mode, WASD moves the camera.
- In Play Mode, WASD controls the character.
- In Play Mode, WASD defaults to run movement.
- In Play Mode, holding Shift switches movement to walk.
- Space is context-sensitive:
  - Animation Preview Mode: Play/Pause.
  - Play Mode: Jump.
- MVP jump trajectory is authoritative from PreviewRuntime character physics using character tuning such as jump height and gravity.
- Jump animation selection/synchronization is layered on top of the preview physics trajectory.
- MVP jump behavior includes the GDD grace-window rule:
  - Standing neutral jump starts as `jump`.
  - If directional input arrives within `jumpDirectionGraceWindow`, the same `jump` role can be paired with trajectory analysis/tuning.
  - Once committed, jump direction stays locked until landing.
- MVP enforces strict no-air-steering after takeoff until landing.
- MVP Play Mode includes coyote time and jump buffer using character tuning defaults.
- MVP landing severity follows the GDD fall height/time rules:
  - Below `landSoftHeight`: `land`.
  - Between `landSoftHeight` and `landMedHeight`: `land`.
  - Between `landMedHeight` and `maxSafeFallHeight`: `land`.
  - Landing while still committed to forward velocity can use `land` plus motion analysis/tuning.
  - Above `maxSafeFallHeight` or `maxSafeFallTime`: unsafe/death condition; preview can reset or show a warning instead of implementing death presentation.
- MVP Play Mode includes a Drop Test control to lift/drop the character from a chosen height for landing validation.
- Drop Test has quick controls in the viewport toolbar and exact settings in the inspector.
- Play Mode camera defaults to an orbit/follow hybrid:
  - Camera target follows the character.
  - User can orbit and zoom.
  - A reset-behind-character action can be added.
- Entering Play Mode resets the character to the origin on the preview plane.
- MVP Play Mode uses an infinite ground plane at `Y = 0`.
- Future preview automation may add paths/points for automated movement tests.
- Future clip recording should support an editor-authored capsule/controller track: the user places a capsule, records or keys its motion on the timeline, then bakes/saves that recorded motion as a new clip for preview and later database use. This is an authoring workflow, separate from importing FBX/BVH source clips.
- MVP includes jump.
- Play Mode is a mechanics prototype, not only an animation viewer.
- Studio PreviewRuntime can become the reference gameplay controller if it produces better behavior than the current game controller.
- The tool should not blindly mimic an existing controller when a cleaner preview/runtime model reveals better mechanics.
- Jump should go through real motion-matching constraints and clip-role requirements as early as possible.
- Missing required roles do not block Play Mode; a warning strip shows degraded/incomplete behavior.

The character tuning inspector shows `stopDistAt100` as the editable designer knob and displays derived read-only values such as run deceleration, stop time, and stop distances.

The inspector should show derived stop previews, including run stop distance/time and walk stop distance/time.

The character tuning inspector also shows derived jump values such as apex time and level-ground airtime from `jumpHeight` and `gravity`.

Stop/deceleration model comes from the GDD locomotion spec:

- Designers tune `stopDistAt100`.
- Run deceleration is derived as `runSpeed^2 / (2 * stopDistAt100)`.
- At defaults, `3.0^2 / (2 * 0.56) ~= 8.04 m/s^2`.
- `stopDistAt100` is the source-of-truth stopping distance from maximum run speed.
- Stop distance scales with speed squared.
- Stop clips at sub-max speed use partial playback until velocity reaches zero.

- `Free3D`
- `SideScrollerXY` (Z locked)
- `TopDownXZ` (Y vertical/ground plane)
- `Custom`

Default movement plane mode is `Free3D`.

Play Mode input direction:

- `Free3D`: WASD is relative to camera forward projected onto the ground plane.
- `SideScrollerXY`: A/D moves along X, Space jumps along Y, Z is locked.
- `TopDownXZ`: WASD moves on XZ ground plane; exact camera/world-relative behavior can be refined later.
